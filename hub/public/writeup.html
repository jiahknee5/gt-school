What I built

A centralized hub for GT Anywhere's marketing functions, standing on one load-bearing backbone: program-isolated Postgres (row-level security), field-directional CRM↔app sync with idempotent payments and a parity / data-confidence signal, and app-level RBAC. On top sit three deep modules — CRM Ops, Budget, Decision Queue — a Status command center with a checked AI layer, and one live end-to-end campaign, the GT Challenge: a real lead flows ad → quiz → a $100 Stripe test deposit → tracked through to HubSpot. Real code where it matters; every stand-in is labeled.

What I went deep on — vs. stubbed — and why

Two days isn't enough for 13 modules, and trying is the red flag. I built a coherent vertical slice and stubbed the rest deliberately.

The backbone (Phase 1)

  • Isolation, proven — program-scoped tables behind RLS + FORCE; the app runs as a least-privilege role (app_rw, NOSUPERUSER / NOBYPASSRLS). An unset program scope returns 0 rows (fail-closed); a cross-program write is rejected by the database, not policed by app code.
  • Idempotent payments — a re-delivered Stripe event is recorded once (processed_events); balances are monotonic. The live demo runs a real $100 test deposit through the same handler the webhook uses.
  • Bidirectional sync + parity — field-directional authority decides who wins per column; when Supabase and HubSpot disagree, parity drops and the data-confidence banner fires across the HubSpot modules.

Key decisions & trade-offs

  • AI grader behind a swappable interface, with a rules floor — quality from the model, determinism for the demo, and capture never blocks on a flaky API. (over: a raw LLM in the request path, or a pure rules grader.)
  • The LLM writes prose, never numbers — every figure is pulled from data against a rubric; the LLM rewrite ships only if it still passes conformance. (over: letting the model state the numbers.)
  • One definition per number — CPQL is measured (spend ÷ qualified), the budget total computed in one place. (over: hard-coding an impressive headline.)
  • Security fails closed — role resolved server-side from the verified session; unset scope → 0 rows; least-privilege DB role. (over: trusting a role claim from the request, or the service-role key on program paths.)
  • No "not gifted" bucket — the quiz is a fit screen for a child, never a verdict.
  • Honesty over theater — real vs stand-in is labeled everywhere; gaps are tracked, not faked green.

Where I honored / bent the spec

  • Honored — source-of-truth (funnel/income = app_form, engagement = HubSpot, budget = Hub); known-broken surfaced honestly (UTM kept as (not set), GA4 flagged low-confidence).
  • Bent — the demo link runs dev-auth (pick a role, no password) so reviewers can switch roles — but the role is server-enforced from the signed session; one env var swaps in a real IdP.

Appendix B

The AI layer — useful, and checkable

  • What's AI vs not — the Status verdict and Ask-the-Hub answers are LLM prose; every number is pulled from the database. The LLM never invents figures.
  • Generated against a rubric — each claim has a rubric (e.g. "north-star deposits must cite Supabase and be a running sum as of the reporting week, never the full-dataset total"). The deterministic draft is the floor; an LLM rewrite ships only if it still conforms.
  • Agent graph (Ask-the-Hub) — validate request → de-identified role-scoped snapshot → policy check → classify agent → intent → retrieve-with-citations → synthesize → compose → output scan (no PII). Every retrieval carries citations.
  • Evals + provenance — per-node eval rows (input / expected / actual / pass) at /dev/agents; per-claim source + self-eval at /dev/rubrics. Runs deterministically in CI — no model or network needed.
Appendix C

Data & schema — honest by construction

Sources

  • Real: HubSpot · Stripe · Supabase · Open Data (TX).
  • Stand-in (labeled): Meta · GA4 · summer.gt · X — behind the same interfaces, each row tagged _standIn + _source.

Join keys

  • utm_campaign — attribution.
  • match_key — dual-source reconciliation (counted once).

Model

family (+children) → program (fall · summer, RLS-isolated) → enrollment → payment. Ledgers: processed_events, sync_outbox, decision_event (append-only).

Reproducible & stressed

  • Deterministic seed → reset to a known state.
  • 15 deliberate edge cases (duplicate payment, family in two programs, conflicting records, missing UTM…) so isolation & reconcile are provable.
Appendix D

Verification

  • Pure CI gate — npm run verify = build + lint + the test suite, with no external services (deterministic seed + mocked SQL). This is the always-green gate.
  • Credential-gated — live-integration tests (reconcile, payments, webhook, parity, real DB connection) run only when Supabase / HubSpot / Stripe keys are present, and skip cleanly otherwise.
  • Brief-to-test map — /dev/tests maps each "prove it" scenario to the test that proves it; /dev/test-suite is the full inventory by category.
