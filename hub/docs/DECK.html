03The lens · the marketing leader's real problem

The risk wasn't a missing feature — it was that marketing couldn't trust its own numbers across two programs and many channels

05Judgment · calls I made on purpose

Where the spec left a choice, I made it deliberately — AI that can't degrade trust, one definition per number, security that fails closed

Decision I made on purposeWhyWhat I rejected
AI grader behind a swappable interface — record-replay + rules floorassessment needs the model's quality, the demo needs determinism, capture can't block on a flaky APIa raw LLM in the request path; a pure rules grader
The verdict's LLM rewrite is eval-gatedthe model can sharpen the language but never degrade the number — the deterministic draft is the floor, the rewrite passes only if it still conformsletting the model write the exec verdict directly
One definition per number; CPQL measured"a number means the same thing everywhere" is the spec's hardest rule; a faked headline is the easiest tellhard-coding an impressive CPQL constant
Security fails closed — role from the verified user, unset scope → 0 rows, least-privilege DB rolea leak should be impossible by construction, not policed after the facttrusting a role claim from the request; service-role key on program paths
No "not gifted" bucket — lowest result is "explore"it's a fit screen for a child, not a verdict — never gate a kid outa pass/fail grader

The rule under all five: honesty over polish. Stand-ins labeled, gaps tracked, the decision audit append-only by DB grant — never faked green.

07The data behind the dashboards

The dashboards sit on 1,200 families modeled to GT's real funnel — channel behavior a marketer recognizes, every stand-in labeled

for marketing

The numbers behave like the real funnel — so the reconciliation, pacing, and CAC math are actually exercised, not happy-path.

  • Modeled, not lorem-ipsum — X converts ~45%, Facebook is a volume trap at ~8%, a clicked lead converts 53% vs 11% cold; 1,200 families across stages and time.
  • 15 deliberate edge cases — duplicate payment, a family in two programs, conflicting CRM-vs-app, missing UTM — so isolation & reconcile are provable against data built to break them.
  • Reproducible — deterministic seed; npm run reset to a known clean state.

A2Appendix · click-by-click demo

Click-by-click — the live walkthrough, in order (L = live URL · D = local + seeded DB)

  1. Sign in L — gt-school-hub.vercel.app → /login → pick Leader (dev-auth, no password).
  2. Payment propagates D — run stripe-service.mjs + stripe listen; node pay.mjs <deal> → ledger → program store → HubSpot closedwon. Run twice → no-op (same id). Show /dev/payments.
  3. No leak D — psql as app_rw: SET app.current_program='fall_enrollment' → only fall; unset → 0 rows; cross-program INSERT → RLS reject.
  4. Role gate L — Leader /m/decisions renders; Operator /m/decisions → /forbidden; open /api/decisions → 403 JSON (Network tab).
  5. Budget reconciles L — /m/budget → $365,000 / 4 workstreams; variance flag “Guerrilla 12% over plan.”
  6. Confidence banner L — /m/nurture → banner “overall 92.76%.” (Edit→drop toggle is D / tested.)
  7. Citations L · admin — /m/dashboard Source column → click ⛁ Supabase on Applicants → /dev/integrations#supabase_app_form highlights.
  8. GT Challenge L — incognito /gifted-quiz → consent → submit → bucket; /m/gt-challenge KPI ($8,208 · 12 · $684); /m/submissions read-back is D.
  9. Open Data flips a decision L — /m/decisions $18K guerrilla → “pilot → approve” ($41 vs $63); failure path = degrade-to-fixture (opendata.test.ts).
  10. Do NOT click Approve/Reject on a queue card — seed-rendered ids ≠ DB ids → 404/500. Demo the ruling + audit via tests/decisions.test.ts.
